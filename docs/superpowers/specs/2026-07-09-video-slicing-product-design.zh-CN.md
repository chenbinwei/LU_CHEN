# AI 自动切片产品设计文档

日期：2026-07-09

状态：待确认草案

项目：LU_CHEN

## 1. 产品定位

这个产品是一个面向影视解说、短剧解说、内容矩阵号运营者的 AI 成片工具。

它的核心不是简单地“把长视频剪短”，而是帮助用户把一个原始视频素材快速变成多个可以发布的解说成片，减少写文案、剪辑、配音、字幕、转格式这些重复工作。

核心价值：

- 节省内容生产时间。
- 让一个素材可以快速生成多个版本。
- 帮助不会写文案、不会剪辑的用户做出影视解说视频。
- 支持矩阵号测试，比如不同时间长度、不同声音、不同 BGM、不同字幕语言、不同画幅。

第一版只考虑中文解说。多语言配音暂时不做。

## 2. 目标用户

第一批目标用户不是成熟的专业剪辑师，而是想成为影视解说或短剧解说创作者的人。

典型用户：

- 想做影视解说、短剧解说、剧情解说内容。
- 手里有视频素材，但缺少文案和剪辑效率。
- 想用同一个素材测试多个账号、多个版本。
- 希望用产品界面完成流程，而不是命令行。
- 商业版中不希望自己配置模型 API Key。

## 3. MVP 范围

第一版建议做成本地单人产品体验，同时在数据结构上预留未来商业化能力。

MVP 包含：

- 一个项目对应一个原视频素材。
- 一个项目下面可以有多个版本。
- 项目层保存转写字幕、用户补充信息、视频基础理解。
- 版本层保存文案、目标时长、声音、BGM、字幕、画幅和最终导出结果。
- 生成完整文案预览。
- 用户可以不编辑文案，直接生成成片。
- 用户也可以编辑完整文案，再生成成片。
- 用户编辑文案后，系统自动重新分段、重新匹配画面。
- 支持 AI 配音。
- 支持 Fish Audio 声音克隆选项。
- 支持可选 BGM。
- 支持调节配音速度。
- 支持调节配音音量。
- 支持调节 BGM 音量。
- 支持两种音频模式：纯解说、关键原声。
- 支持字幕烧录和 SRT 导出。
- 支持中文字幕、英文字幕、中英双语字幕。
- 支持原画幅和竖屏 9:16 背景填充。
- 最终视频时长需要尽量贴近用户输入的目标时长。

MVP 不包含：

- 登录。
- 支付。
- 订阅扣费。
- 多用户项目隔离。
- 云端存储。
- 完整云端渲染。
- 多素材批量队列。
- 智能主体跟随裁切。
- 多语言配音。
- 专业时间线编辑器。
- 用户手动逐段换画面。
- 成片前展示复杂的画面匹配结果。

## 4. 产品流程

第一版应该是一个引导式工作流：

1. 用户上传一个原始视频。
2. 系统提取音频，并生成带时间戳的转写字幕。
3. 用户填写轻量补充信息：
   - 作品名或主题。
   - 主要人物。
   - 剧情或事件梗概。
   - 必须讲到的点。
   - 明确不要出现的内容。
4. 用户在项目下创建一个版本。
5. 用户配置版本参数：
   - 目标时长。
   - 音频模式。
   - 声音克隆。
   - BGM。
   - 配音速度。
   - 配音音量。
   - BGM 音量。
   - 字幕语言。
   - 成片画幅。
6. 系统生成完整的不分段文案预览。
7. 用户选择直接使用文案，或手动编辑完整文案。
8. 系统自动把最终文案分段。
9. 系统把文案段落匹配到原视频字幕时间戳和画面区间。
10. 系统生成 TTS 配音。
11. 系统剪辑画面、处理原声策略、混合 BGM、烧录字幕并导出视频。
12. 用户下载最终视频和 SRT 字幕文件。

第一版中，用户只负责审文案，不查看复杂的画面匹配结果。

## 5. 项目和版本模型

产品需要区分项目层和版本层。

### 项目

一个项目代表一个原视频素材。

项目层数据：

- `user_id`：为未来账号系统预留。本地 MVP 中可以固定为 `local_user`。
- `project_id`
- 原视频路径或素材 ID。
- 提取出的音频路径。
- 带时间戳的转写字幕。
- 原视频总时长。
- 用户补充信息。
- 可选的视频基础理解或摘要。
- 创建时间。
- 更新时间。

### 版本

一个版本代表这个素材的一套成片方案。

版本层数据：

- `version_id`
- `project_id`
- `parent_version_id`：为复制版本预留。
- `generation_group_id`：为未来一键生成多个版本预留。
- `variant_goal`：为未来自动生成不同版本目标预留。
- 目标时长。
- 音频模式。
- 声音克隆 ID。
- BGM ID 或路径。
- 配音速度。
- 配音音量。
- BGM 音量。
- 字幕语言。
- 成片画幅。
- 初版生成文案。
- 用户确认后的最终文案。
- 分段后的脚本。
- 画面匹配结果。
- TTS 音频路径。
- 字幕文件路径。
- 导出视频路径。
- 任务状态。

这个结构可以支持当前的手动多版本，也能为未来自动批量生成多版本做准备。

## 6. 单素材多版本策略

MVP 支持一个素材生成多个版本，但不做多素材批量队列。

第一版行为：

- 用户手动新建版本。
- 用户可以复制已有版本。
- 用户手动修改目标时长、声音、BGM、字幕、画幅等参数。
- 每个版本可以独立生成自己的文案和最终视频。

未来预留：

- 一键生成多个版本。
- 按目标时长自动生成多个版本。
- 按讲述角度自动生成多个文案版本。
- 按声音、BGM、字幕语言、画幅自动生成多个版本。
- 通过 `generation_group_id` 管理同一批生成任务。

## 7. Context 设计

`context.json` 不应该变成给普通用户看的复杂 prompt 配置文件。

它应该分成两层。

### 系统上下文层

由开发者或高级运营者维护。

它定义：

- 内容类型。
- 文案写作规则。
- 什么信息可以合理补全，什么不能编。
- 文案质量约束。
- 如何讲一个完整故事。
- 如何避免编错剧情。
- 如何处理人物名和人物关系。
- 如何选择关键原声片段。
- 当前内容分支下的叙事结构。

系统层可以继续使用 branch，但 branch 表示的是内容生产模式，不是简单的“风格”。

示例：

- `film_commentary.classic_scene`
- `film_commentary.plot_recap`
- `film_commentary.character_focus`
- `short_drama.recap`

### 用户补充信息层

前端以轻量表单形式展示给用户。

字段：

- 作品名或主题。
- 主要人物。
- 剧情或事件梗概。
- 必须讲到的点。
- 明确不要出现的内容。

这一层很重要，因为只靠字幕经常不够。比如原视频台词里可能没有说出人物名字，但影视解说文案必须知道这个人是谁。

## 8. 文案流程

文案属于版本层。

系统首先生成一篇完整的不分段文案预览。第一版不展示内部的分段结果和画面匹配结果。

文案生成要求：

- 文案要能讲一个完整故事。
- 文案要适合真人口播式影视解说。
- 不能只是原字幕摘要。
- 当字幕信息不足时，要合理使用用户补充信息。
- 避免错别字、错词、英文残留和无关词。
- 遵守用户填写的禁止内容。
- 文案长度要服务于目标成片时长。

用户确认或编辑文案后：

1. 系统自动把完整文案分段。
2. 每段文案生成时长估计。
3. 每段文案匹配原字幕和原视频证据。
4. 根据真实 TTS 时长生成最终配音时间轴。
5. 最终视频时间轴围绕确认后的文案和目标时长构建。

用户编辑的是故事文案，不是时间轴。

## 9. 时长规则

目标时长永远表示最终导出视频的总时长。

比如用户输入 120 秒，最终成片就应该尽量接近 120 秒。这个总时长包含：

- AI 解说配音。
- 关键原声片段，如果开启。
- 必要的停顿或转场。
- BGM 铺底。

系统需要校验：

- 目标时长必须大于 0。
- 目标时长必须小于原视频时长。
- 时长容差必须大于等于 0。
- 最终导出视频应该尽量落在配置的容差范围内。

如果生成的配音太短或太长，系统应该通过以下方式处理：

- 必要时重新生成或修正文案长度。
- 在合理范围内调整 TTS 速度。
- 根据真实音频时长重新构建画面选择。

产品需要避免用户输入 100 秒，最后却生成 60 秒成片的情况。

## 10. 音频模式

MVP 只有两种音频模式。

### 纯解说

行为：

- 删除原视频声音。
- 使用 AI 配音作为主音频。
- 可选混合 BGM。
- 烧录生成字幕。

这是最稳定的模式，也应该作为兜底方案。

### 关键原声

行为：

- 大部分时间由 AI 解说讲故事。
- 系统自动从原视频中选择关键原声片段。
- 原声出现时，AI 解说暂停。
- BGM 可以继续存在，但需要压低。
- 原声只能来自原视频真实时间戳。

规则：

- 关键原声占最终视频总时长的 10%-20%。
- 每段关键原声一般控制在 2-8 秒。
- 优先选择冲突落点、名台词、情绪爆点、人物转折。
- 如果找不到可靠原声片段，自动退回纯解说逻辑。
- 不能凭空生成原声，也不能编造原片台词。

比如 120 秒视频中，关键原声通常占 12-24 秒。

## 11. BGM

BGM 是可选能力。

MVP 控制项：

- 选择 BGM。
- 关闭 BGM。
- 设置 BGM 音量。

混音要求：

- BGM 要能听见，能够支撑情绪。
- BGM 不能压过解说配音。
- BGM 开头不应该默认出现过长、过轻的淡入。
- 关键原声模式下，BGM 需要在解说和原声下面适当降低。

未来扩展：

- BGM 分类。
- 根据内容分支自动推荐 BGM。
- 不同段落使用不同 BGM。
- 第一版不需要做卡点剪辑。

## 12. 声音和 TTS

MVP 支持 Fish Audio 声音克隆选项。

用户可见控制：

- 选择声音克隆。
- 设置配音速度。
- 设置配音音量。

技术要求：

- 声音克隆 ID 应该作为可复用声音资产保存。
- 声音设置属于版本层。
- TTS 生成后必须测量真实音频时长。
- 视频时间轴要基于真实 TTS 时长重新构建。

商业版行为：

- 用户不需要提供 Fish Audio 或其他服务的 API Key。
- 平台统一管理模型和 TTS 服务。

本地开发行为：

- API Key 可以放在本地 `.env`。
- `.env` 不能提交到 Git。
- `.env.example` 只保留安全的变量示例。

## 13. 字幕

字幕应该来自最终解说文案，而不是原视频字幕。

原因：

- 最终字幕必须跟随新的 AI 配音时间轴。
- 原视频字幕时间戳主要用于画面匹配，不用于最终字幕显示。

MVP 字幕输出：

- 中文字幕烧录视频。
- 英文字幕烧录视频。
- 中英双语字幕烧录视频。
- 中文字幕 SRT。
- 英文字幕 SRT。

素材建议：

- 建议用户上传无内嵌字幕的视频。
- 如果原视频本身已经有字幕，系统生成字幕可能和原字幕重叠。
- OCR 清理或去除原字幕不属于 MVP。

## 14. 画幅

MVP 支持两种导出画幅。

### 原画幅

行为：

- 保持原视频比例。
- 适合预览、横屏平台或质量保守导出。

### 竖屏 9:16 背景填充

行为：

- 生成竖屏 9:16 视频。
- 中间保留完整原画面。
- 背景使用同一画面的放大模糊版本填充。
- 字幕放在竖屏安全区域。

MVP 不做智能主体跟随裁切。这个可以作为后续高级功能。

## 15. 前端方向

推荐技术栈：

- React
- TypeScript
- Vite

前端应该是引导式创作流程，不是专业剪辑时间线。

主要页面：

- 项目列表。
- 新建项目和上传视频。
- 项目详情。
- 用户补充信息表单。
- 版本列表。
- 版本参数配置。
- 文案预览和完整文案编辑。
- 生成进度。
- 导出和下载页面。

MVP 不应该暴露复杂的 prompt 配置、分段匹配、时间线编辑等内部细节。

## 16. 后端方向

推荐后端技术栈：

- FastAPI 作为未来产品 API。
- 现有 Python pipeline 作为处理核心。
- FFmpeg 做音视频处理。
- Faster-Whisper 或类似工具做转写。
- DashScope/Qwen 或其他 LLM 服务做文案生成和文案处理。
- Fish Audio 做 TTS 和声音克隆。

即使是本地模式，后端也应该逐步抽象出项目、版本、任务这些概念。

未来 API 形态可以是：

- `POST /projects`
- `GET /projects`
- `GET /projects/{project_id}`
- `POST /projects/{project_id}/versions`
- `POST /versions/{version_id}/script`
- `PUT /versions/{version_id}/script`
- `POST /versions/{version_id}/render`
- `GET /jobs/{job_id}`
- `GET /versions/{version_id}/exports`

这些 API 不需要马上实现，但代码结构应该朝这个边界演进。

## 17. 任务和状态模型

成片生成应该被建模为任务。

常见任务状态：

- `pending`
- `running`
- `failed`
- `done`
- `cancelled`

常见任务阶段：

- `extract_audio`
- `transcribe`
- `generate_script`
- `segment_script`
- `match_visuals`
- `generate_tts`
- `fit_duration`
- `cut_video`
- `mix_audio`
- `burn_subtitles`
- `export`

这样后面接前端时更容易展示进度，也更方便排查问题。

## 18. 商业模式预留

计划中的商业模式是订阅加额度。

商业版用户只需要使用产品，不需要配置自己的 API Key。

推荐额度单位：

- 最终生成视频分钟数。

可以附加的额度维度：

- 导出次数。
- 声音克隆数量。
- 并发生成任务数。
- 存储时长。
- 高级声音、高级 BGM、高级批量能力。

登录、支付、订阅扣费可以后面再做，但数据结构需要预留：

- `user_id`
- `duration_seconds`
- `job_id`
- `voice_clone_id`
- 导出记录。

## 19. 后续扩展

未来可以扩展：

- 多素材批量队列。
- 一键生成多个版本。
- 多语言配音。
- 智能竖屏主体跟随裁切。
- 分段级预览。
- 分段级画面替换。
- 团队空间。
- 云端渲染。
- 平台导出预设。
- 自动生成标题、封面、简介。
- 发布平台集成。
- 版本效果数据分析。

这些都不应该阻塞 MVP。

## 20. 关键风险

### 文案质量风险

产品高度依赖文案质量。文案必须像真人影视解说，而不是机械字幕摘要。

缓解方式：

- 保留轻量但必要的用户补充信息。
- 使用系统上下文分支控制不同内容生产模式。
- 在最终成片前提供完整文案预览和编辑。

### 画面匹配风险

如果文案和画面匹配不好，成片会显得脱节。

缓解方式：

- 文案段落只能匹配到原字幕和原视频证据。
- 内部保留匹配置信度。
- 置信度低时回退到更保守的顺序匹配。

### 时长控制风险

如果目标时长不准，产品对矩阵号用户就不可靠。

缓解方式：

- 校验目标时长必须小于原视频时长。
- 测量真实 TTS 时长。
- 根据真实音频时长重建视频时间轴。
- 设置最终导出时长容差。

### 关键原声风险

自动选择原声可能选到不够精彩或不够清晰的片段。

缓解方式：

- 原声总比例限制在 10%-20%。
- 原声片段尽量短。
- 原声必须有原字幕或原视频时间戳依据。
- 置信度低时退回纯解说。

### 版权和平台风险

影视剧和短剧素材可能涉及版权问题。

缓解方式：

- 产品设计上不要鼓励侵权使用。
- 商业化时需要用户责任条款、内容政策和下架处理机制。

## 21. MVP 验收标准

MVP 可以认为成功的条件：

- 用户可以从一个原视频创建项目。
- 系统可以完成原视频转写。
- 用户可以填写轻量补充信息。
- 用户可以在同一个项目下创建多个版本。
- 每个版本可以设置独立的目标时长、音频模式、声音、BGM、字幕和画幅。
- 系统可以生成完整文案预览。
- 用户可以编辑完整文案。
- 系统可以基于最终文案自动分段并匹配画面。
- 系统可以生成配音和最终视频。
- 最终视频时长接近用户输入的目标时长。
- 系统可以导出烧录字幕视频和 SRT 字幕文件。
- 同一个原视频至少能生成两个有明显差异的版本。

## 22. 自检

这个设计把 MVP 控制在可实现范围内：

- 暂时不做云端渲染、支付、登录和多素材批量。
- 暂时不做时间线编辑和分段级画面预览。
- 用户只关注文案和生成参数，降低使用门槛。

这个设计也为后续产品增长预留了空间：

- 项目、版本、任务结构可以接前端和 SaaS。
- 版本模型可以支持矩阵号多版本生产。
- 音频模式可以逐步接近真实影视解说效果。
- 字幕和画幅设置适合短视频平台使用。

仍然需要后续确认的问题：

- 第一批应该实现哪些 `context` 分支？
- 生产环境中最终时长容差应该设置为多少？
- 画面匹配置信度低于多少时需要回退？
- 未来商业版中，失败导出如何返还额度？
- 关键原声模式是否应该在 MVP 中默认开放，还是先作为实验功能？
